0项目背景与策略锚点PRD §1-2

业务背景

钱包内提现用户主要现金来源为返现。提现体验作为返现尾程峰终体验，间接影响返现用户 LTV。

  • 钱包有余额 + 有返现订单用户占比约 18.43%
  • 钱包资金流水中返现业务占比 94.60%

用户痛点（客诉 TOP 标签）

  • 无法提现/提现受阻 27.5%
  • 提现失败/退回 24.8%
  • 提现未到账/处理中 21.1%
  • 账号绑定/身份不符 8.8%
30% 提现客诉直接涉及资金，55% 涉及到账焦虑。

优化策略锚点（对应本次需求方案）

① 状态流程：缩短"处理中"停留时间 + 超时自动排查/补偿
② 通知完备：成功/失败结构化通知，明确失败原因
③ 余额逻辑：到账前不扣减，新增"冻结"态
④ 余额明细：明确余额来源
⑤ 过程可视：提现进度模块可视化
⑥ 历史信任：余额变动列表展示历史到账时长
1金额字段定义说明PRD §4.4 表1
字段定义说明 / 取值
账户总额用户当前在美柚平台钱包账户中，所有资金状态的总和账户总额 = 流动金额 + 冻结金额
流动金额账户总额中，无任何限制、可正常操作提现的金额部分对应「可提现余额」展示数值，是「提现」按钮的校验上限
冻结金额账户总额中，因提现处理中等原因，不可操作提现的金额对应「重复操作」提示中的 {冻结金额}元提现中
⚠ 核心策略变更：到账成功前不做余额扣减（资金从「流动」改为「冻结」），用户感知账户总额不变。
2钱包页 · 提现操作模块 — 初始态PRD §4.4 提现操作模块
场景：用户初次进入钱包页，无在途提现
可提现余额
  • 固定文案：可提现余额（元）：¥{账户总额}
  • 取值：取「账户总额」字段数值（即流动+冻结之和）
提现金额输入框
  • 默认不填入数值，由用户手动输入或点击「全部提现」回填
「提现」按钮状态机
场景条件按钮表现
正常发起输入金额 ≤ 流动金额 且 无在途蓝色激活 → 点击进入「提现申请成功弹窗」
不可点击输入框无数值置灰不可点击
重复操作已有提现处理中 + 点击吐司「有一笔提现处理中」+ 下方提示
余额不足输入金额 > 流动金额下方红字「输入金额超过可用余额」+ 按钮置灰
「全部提现」按钮
  • 展示条件：无在途提现时展示；有在途则隐藏
  • 点击规则：点击后将「流动金额」回填至输入框
2钱包页 · 提现操作模块 — 重复操作（在途处理中）PRD §4.4 场景三
场景：当前已有一笔提现处理中
输入态
  • 输入框置灰不可输入
  • 提现按钮替换为灰色禁用按钮 一笔提现处理中
  • 「全部提现」按钮隐藏
提示文案
  • 输入框下方固定文案：{冻结金额}元提现中，请勿重复操作。
  • 若用户尝试再次点击提现按钮：弹出 Toast 有一笔提现处理中
提现进度模块联动
  • 提交申请成功后，关闭「提现申请成功弹窗」时同步在提现操作模块下方挂载「提现进度模块」
  • 该状态下「可提现余额」=「流动金额」（冻结金额不展示在该数字上但仍属于账户总额）
2钱包页 · 提现操作模块 — 输入超额PRD §4.4 场景四 · 6月1日update
场景：用户输入金额 > 流动金额
校验规则
  • 触发条件：用户输入金额数值 > 「流动金额」
  • 表现：
    1. 输入框下方红字提示：输入金额超过可用余额
    2. 「提现」按钮置灰不可点击
  • 当用户修正金额至 ≤ 流动金额，自动恢复正常态
2钱包页 · 提现操作模块 — 未绑定收款账号演示补充 · PRD 未细化

4提现进度模块 — 场景一：提现处理中PRD §4.4 提现进度模块
场景：发起提现申请，尚未获得明确到账结果
模块位置与展示规则
  • 位置：固定展示在「提现操作模块」下方
  • 手动关闭：点击右上角 × → 该模块隐藏
  • 自动关闭：距该笔提现创建时间 ≥ {x} 天 → 不展示（x 取决于用户提现后页面复访时机，待评审定）
基础信息板块（同弹窗）
  • 4 个字段：提现金额 / 提现渠道 / 收款账户 / 创建时间
  • 支持「详情 ▲▼」折叠详情信息板块
详情信息板块（3 节点）
节点定义固定文案副文案
①用户点击「提现」按钮且申请成功，生成提现申请记录发起提现申请取 提现申请成功弹窗的创建时间（秒）
②从返现调用三方接口，将提现申请发送至三方，待三方返回结果{提现渠道}处理中取 调用三方接口的时间（秒）；若无则不展示
③三方渠道明确返回到账结果（成功/失败）的节点到账成功当前进度=三方处理中 → 展示 {提现渠道}处理中，请耐心等待

5余额变动模块（钱包页）PRD §4.4 余额变动模块
场景：钱包页下方固定区，展示历史变动
位置与展示条件
  • 位置：固定展示在「提现进度模块」下方
  • 规则：用户无历史余额变动 → 不展示该模块
  • 余额变动定义：账户余额发生实际金额变化时记一次（失败未动账的提现不进入该列表）
模块头部
  • 左侧标题固定文案：余额变动
  • 右侧入口固定文案：全部 ›，点击跳转「收支明细页」（见 §6）
列表展示规则
  • 展示条数：一行一条，最多 3 行
  • 排序：按变动时间倒序，取最新 3 条
单条信息区
字段规则
记录标题 收入（到账）：返现订单 → 返现订单（{订单标题}）；活动 → {活动名称}
支出（提现）：成功提现至支付宝/微信 → 提现至支付宝 / 提现至微信
变动时间戳取 收支明细变动时间，{yyyy-mm-dd hh:mm:ss}
变动金额 收入：+¥{到账金额}（粉色）
支出：-¥{提现成功金额}（深色）
补充信息 支出：到账时长 = 提现成功时间 − 提现申请时间（统一换算为秒）
收入：若该笔返现有补贴 → 展示「含补贴X元 / 红包X元」；无则留空
6收支明细页演示补充 · PRD 仅引用入口
入口：钱包页右上角「···」 → 「收支明细」
或：余额变动模块右上角「全部 ›」
页面结构
  • 顶部 Tab：现金 / 金豆 / 柚币（本期仅"现金" Tab 上线，其余 Tab 提示「暂未开通」）
  • 分组头：「本月」（后续可扩展按月份分组）
  • 列表项规则参考 §5「单条信息区」表格
点击行为
  • 点击提现条目 → 弹出底部半弹窗，展示该笔提现详情（结构 = §3 提现申请成功弹窗 / §4 进度模块）：
    • 处理中 → 显示基础信息 + 3 节点进度（节点 2 进行中）
    • 成功 → 显示基础信息 + 3 节点全完成
    • 失败 → 显示基础信息 + 失败原因 + 「管理收款账户」
  • 点击返现订单条目 → 跳转对应订单详情页
  • 点击活动/红包条目 → 暂跳转活动详情页（评审待定）
7账户绑定流程演示补充 · PRD 未细化
入口：钱包页收款账号入口（未绑定时）
页面结构
  • 渠道选择：支付宝 / 微信（互斥单选）
  • 账号输入：根据所选渠道，要求输入对应绑定手机号
  • 提示文案：请确保{渠道}账号已完成实名认证，且姓名与美柚实名一致。
  • 「下一步」按钮：输入达到 11 位后激活 → 弹出二次确认弹窗 → 确认后回钱包页并 Toast「绑定成功」
8收款账号详情页演示补充 · PRD 仅引用入口
