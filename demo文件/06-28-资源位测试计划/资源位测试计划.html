资源位测试计划 · 新方案产品流程图

核心逻辑：同一资源位下，正式投放规则与测试计划并行配置、互不影响。 用户请求资源位时，服务端先执行 P0 测试计划匹配——仅当「计划进行中 + UID 在白名单 + 当前在测试时段」三者同时满足，才返回测试素材并跳过全部正式规则； 否则进入 P1 正式投放匹配。因此测试 UID 在计划有效期内一定能看到测试内容，普通用户永远看不到测试素材。

一、产品架构：配置解耦 + 匹配优先级

运营在后台维护两条独立配置线；服务端按 P0 → P1 顺序匹配，两条线不交叉、不混填。

P0 测试计划（优先）
P1 正式投放
虚线 = 配置同步至匹配引擎
P0 命中条件三者同时满足：① 测试计划状态 = 进行中；② 当前 UID 在该计划白名单内；③ 当前时间在测试时段内
P0 命中效果直接返回该计划的测试素材，不再判断上下线、人群包、实验、版本等任何正式规则
P1 生效范围所有 P0 未命中的请求（含普通用户、测试 UID 计划过期、白名单外 UID）

二、运营配置流程：测试与正式分离

测试 UID 不再写入定向信息；通过独立「测试计划」管理，正式配置全程不动。
