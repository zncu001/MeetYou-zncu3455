V1 vs V2 产品方案对比

1. 核心差异速览

维度 V1 条件组级商城 V2 条件级商城（本 demo）
商城归属层级 条件组 条件
商城选择时机 新增条件组时弹商城选择 Modal 新增条件时，在"限制条件"Modal 顶部 Tab 切换
条件组内字段约束 仅可选择该组绑定商城的字段 同组可混合淘宝和京东字段
嵌套子组 自动继承上层商城 嵌套子组无商城概念
跨商城 AND 限制 有显式阻断（弹窗确认、AND 置灰） 无前置阻断（依赖保存校验 / 服务端拆分）
条件行视觉标识 条件组带商城徽标 每条条件行左侧带「淘」/「京」徽标
服务端拆分难度 清晰：直接按条件组拆 API 调用 复杂：需把同组内不同商城条件拆开后再合并

2. V2 优劣势

✅ 优势 ❌ 劣势
  • 用户决策更轻：少了"新增条件组时先选商城"这一步
  • 布局更扁平：所有条件视觉上在一个组内，无嵌套层级
  • 更接近多平台并行筛选的直觉（"我想要符合 A B C 这一堆条件的商品，不管哪个商城"）
  • 语义陷阱：同组 AND 「淘宝商品类型=聚划算 && 京东经营方=自营」永远命中为空，但 UI 不阻断
  • 同名字段易混淆："到手价"在淘宝和京东都有，但 key 不同，需要带徽标区分
  • 服务端执行复杂：必须把同组内的不同商城条件按平台拆开调 API 后再合并，逻辑负担上移到服务端
  • 不便扩展更多平台：增加抖音、拼多多后字段总数膨胀，单 Modal 内 Tab 数量增加，认知负担升高

3. V1 优劣势

✅ 优势 ❌ 劣势
  • 语义严谨：条件组绑定商城，杜绝跨商城 AND 这种"必然空集"的配置
  • 服务端拆分清晰：每个条件组就是一个独立的 API 调用单元
  • 易扩展多平台：新增平台只需扩展 Schema，不影响交互结构
  • 对运营友好：清晰看到"这条规则覆盖哪些商城"
  • 多一步操作：新增条件组时要选商城
  • 跨商城混合时层级感稍重：根组 OR (淘宝组) OR (京东组)
  • 已绑商城的条件组无法直接换商城，需删除重建

4. 应该选哪一版？建议判断标准

如果你更看重…… 推荐
语义严谨、服务端执行清晰、可扩展性 V1 条件组级商城
交互轻量、运营操作快速、单一规则结构扁平 V2 条件级商城
未来还会接入抖音、拼多多、唯品会等更多商城 V1
未来基本只在淘宝 + 京东两端做选品 两者皆可，V2 略快

5. 一个具体例子的对照

需求："我要找婴儿奶粉、到手价 120–2000 元、有可领券的商品，淘宝和京东都要"。

V1 配置结构 V2 配置结构 
根组（OR）
├─ 淘宝条件组（AND）
│   ├─ 三级类目 = 婴儿奶粉
│   ├─ 到手价 ≥ 120
│   ├─ 到手价 ≤ 2000
│   └─ 是否有优惠券 = 是
└─ 京东条件组（AND）
    ├─ 二级类目 = 婴幼儿奶粉
    ├─ 到手价 ≥ 120
    ├─ 到手价 ≤ 2000
    └─ 是否有优惠券 = 正常可领
 
条件组（OR 套两个 AND 子组）
├─ AND 子组
│   ├─ [淘] 三级类目 = 婴儿奶粉
│   ├─ [淘] 到手价 ≥ 120
│   ├─ [淘] 到手价 ≤ 2000
│   └─ [淘] 是否有优惠券 = 是
└─ AND 子组
    ├─ [京] 二级类目 = 婴幼儿奶粉
    ├─ [京] 到手价 ≥ 120
    ├─ [京] 到手价 ≤ 2000
    └─ [京] 是否有优惠券 = 正常可领
视觉层级感强，所属商城一目了然 条件平铺更紧凑，靠徽标识别商城
